### 1. Чому ви обрали шарову архітектуру?

**Відповідь:**
Шарова архітектура забезпечує:
- **Чіткі межі відповідальності** - кожен шар знає про те, що нижче, але не вище
- **Легше тестувати** - кожен шар можна тестувати окремо
- **Переносимість** - можемо замінити реалізацію (JSON → БД) без змін в інших шарах
- **Масштабованість** - просто додавати нові шари (API, WebUI, etc.)

Альтернативи як CQRS або Hexagonal потребували б більше коду для простого проекту.

### 2. Обов'язково ли було 4 проекти (Domain, Application, Infrastructure, Console)?

**Відповідь:**
Так, для демонстрації курсу. Причини:
- **Domain** - чисті сутності без залежностей
- **Application** - бізнес-логіка, не залежить від файлів
- **Infrastructure** - файли, БД, зовнішні сервіси
- **Console** - UI, може змінюватися без змін логіки

На малому проекті можна було б об'єднати, але така структура краще демонструє архітектурні принципи.

### 3. Чому не використали IoC контейнер (Autofac, Ninject)?

**Відповідь:**
На Lab 34-35 IoC контейнер не був пройдений. На Lab 37 ми впровадили базове впорскування залежностей у Program.cs, що достатньо для проекту цього розміру.

Приклад нашого DI:
```csharp
var repository = new FileSystemExpenseRepository();
var service = new ExpenseService(repository);
```

На v1.1.0 планується додати MS.Extensions.DependencyInjection для більшої масштабованості.

### 4. Це Singleton або Transient для сервісів?

**Відповідь:**
- **Singleton** - категорії (не змінюються протягом сесії)
- **Transient** - витрати (новий об'єкт для кожної операції)

В нашому простому DI це явно:
```csharp
var expenseRepository = new FileSystemExpenseRepository(); // Transient
var expenseService = new ExpenseService(expenseRepository); // Singleton

##  Питання про SOLID принципи

### 5. Як ви застосували Single Responsibility?

**Відповідь:**
- **Expense** - тільки валідує дані витрати
- **ExpenseService** - тільки бізнес-логіка
- **FileSystemRepository** - тільки файловий I/O
- **Menu** - тільки UI логіка

Кожен клас має одну причину для змін.

**Приклад:** Якщо змінимо формат JSON, змінюємо тільки `FileSystemRepository`, а не `ExpenseService`.

### 6. Де Open/Closed принцип?

**Відповідь:**
Інтерфейс `IExpenseRepository`:
- **Відкрито для розширення:** можемо додати нові реалізації:
  ```csharp
  public class SqlExpenseRepository : IExpenseRepository { }
  public class MongoExpenseRepository : IExpenseRepository { }
  ```
- **Закрито для модифікації:** `ExpenseService` не змінюється:
  ```csharp
  public ExpenseService(IExpenseRepository repository)
  {
      _repository = repository; // Може бути будь-яка реалізація
  }
  ```

### 7. Liskov Substitution - як ви його перевіряєте?

**Відповідь:**
Обидві реалізації інтерфейсу `IExpenseRepository` повинні бути замінювані:
```csharp
// Спочатку in-memory
IExpenseRepository repo = new InMemoryExpenseRepository();
ExpenseService service = new ExpenseService(repo);

// Потім файлова система - сервіс працює однаково
repo = new FileSystemExpenseRepository();
service = new ExpenseService(repo);
```
Тести перевіряють, що обидві реалізації дають однаковий результат.

### 8. Які інтерфейси? Interface Segregation?

**Відповідь:**
Ми маємо 3 інтерфейси:
- **IExpenseRepository** - синхронні операції (CRUD)
- **IExpensePersistence** - асинхронні операції (планується v1.1.0)
- **ICategoryRepository** - роботу з категоріями

Кожен клієнт залежить тільки від методів, які використовує. Це чітко демонструє ISP.

### 9. Dependency Inversion - чим залежимо?

**Відповідь:**
Залежимо від абстракцій, не від конкретних класів:

```csharp
// ❌ Bad - жорстка залежність
public class ExpenseService
{
    private FileSystemExpenseRepository _repo = new();
}

// ✅ Good - залежимо від інтерфейсу
public class ExpenseService
{
    private readonly IExpenseRepository _repo;
    
    public ExpenseService(IExpenseRepository repo)
    {
        _repo = repo;
    }
}
Це дозволяє мокувати залежності у тестах.

## Питання про тестування

### 10. Чому 78% покриття, а не 100%?

**Відповідь:**
- **Domain layer** - 95% (критичні правила)
- **Application** - 85% (сервісна логіка)
- **Infrastructure** - 80% (файловий I/O складне тестувати)
- **Console** - 60% (UI логіка важко тестувати)

100% непрактично для UI коду. 78% достатньо для курсового проекту та охоплює критичні сценарії.

### 11. Яке середовище для тестів?

**Відповідь:**
- **Unit тести** - xUnit, без залежностей
- **Mocking** - Moq для інтерфейсів
- **Integration тести** - реальні файлові операції з тимчасовою директорією
- **CI/CD** - GitHub Actions, запускає на кожен push

### 12. Як ви тестуєте файлові операції?

**Відповідь:**
```csharp
[Fact]
public void Save_CreatesFile()
{
    // Arrange
    var tempFile = Path.GetTempFileName();
    var repo = new FileSystemExpenseRepository(tempFile);
    var expenses = new[] { expense };
    
    // Act
    repo.Save(expenses);
    
    // Assert
    Assert.True(File.Exists(tempFile));
    // Cleanup
    File.Delete(tempFile);
}
```

Використовуємо `Path.GetTempFileName()` для створення тимчасових файлів.

### 13. Які fault scenarios ви тестуєте?

**Відповідь:**
- **FileNotFound** - завантаження неіснуючого файлу
- **UnauthorizedAccess** - помилка доступу до файлу
- **JsonException** - пошкоджений JSON
- **ArgumentException** - невалідні дані вводу
- **InvalidOperationException** - неправильний стан

Усі обробляються гарно з інформативними повідомленнями.

## Питання про функціональність

### 14. Які use cases ви реалізували?

**Відповідь:**
Три основні use cases з vision.md:

1. **Додавання витрати**
   - Користувач вибирає категорію, вводить суму та опис
   - Система валідує дані та зберігає витрату
   - Дата та час встановлюються автоматично

2. **Перегляд витрат за період**
   - Користувач вибирає місяць та рік
   - Система повертає витрати за обраний період
   - Показується загальна сума

3. **Отримання статистики**
   - Користувач вибирає статистику
   - Система розраховує суми та середні по категоріям
   - Показується глобальна статистика

**Розширення (Lab 35):** Додали пошук, сортування, редагування, видалення.

### 15. Як зберігаються дані?

**Відповідь:**
JSON файл у папці `data/expenses.json`:
- **Формат:** массив об'єктів з id, category, amount, description, date
- **Кодування:** UTF-8 для підтримки українських символів
- **Завантаження:** при старті програми автоматично
- **Збереження:** на кожну операцію додавання/редагування/видалення

Это позволяет переносити дані между поточків та створюувати резервні копії.

### 16. Чи можна додавати власні категорії?

**Відповідь:**
На v1.0.0 - немає. Категорії зафіксовані в коді. Це обмеження для простоти реалізації.

На v1.1.0 планується:
- Категорії зберігатимуться в окремому JSON файлі
- Користувач зможе додавати/редагувати категорії
- Буде валідація унікальності імен

### 17. Чом немає веб-інтерфейсу?

**Відповідь:**
На курсі фокус на архітектурі та логіці, не на UI. Console достатньо для демонстрації функціональності.

Веб-інтерфейс планується на v1.1.0:
- **Blazor Server** для простоти розробки
- Та ж сервісна логіка, нове UI

### 18. Які code smells ви усунули на Lab 37?

**Відповідь:**
1. **Дублювання валідації** → Централізовано в конструкторах
2. **Магічні числа** → Константи та Enum
3. **Довгі методи** → Розділені на окремі класи
4. **Горизонтальні методи** → Явна обробка помилок
5. **Невдалі імена** → Рефакторено на зрозумілі

### 19. Як ви вимірювали покриття?

**Відповідь:**
```bash
dotnet test /p:CollectCoverage=true /p:CoverletOutputFormat=opencover
```

Потім генеруємо HTML звіт:
```bash
reportgenerator -reports:"**/coverage.opencover.xml" -targetdir:"coverage-report"
```

Результат показує 78% line coverage по всіх шарах.

### 25. Як можна розширити проект?

**Відповідь:**
- **Нова функція (Budget)** - додати Domain entity → Сервіс → Repository → UI
- **Новий інтерфейс (Blazor)** - нові контролери, та ж сервісна логіка

Архітектура допускає розширення без руйнування існуючого коду.

## Питання про курс та навчання

### 28. Які теми курсу найбільш складні було реалізувати?

**Відповідь:**
1. **LINQ оптимізація** - важко писати ефективні запити
2. **Обробка помилок файлів** - багато edge cases
3. **Тестування асинхронного коду** - на v1.1.0 будуть складності
4. **Рефакторинг без перекоду** - балансування між質量 та часом

### 29. Який паттерн найбільш корисний?

**Відповідь:**
- **Repository** - дозволяє легко змінювати персистентність
- **Service** - централізує логіку, легче тестувати
- **Result** - явна обробка помилок, функціональний стиль

Result паттерн найбільш корисний - навіть мала помилка обробляється гарно.

### 30. Над якими темами більше всього працювали?

**Відповідь:**
1. **SOLID принципи** - застосовується скрізь, потребує 30% часу
2. **Тестування** - unit + integration = 25% часу
3. **Документація** - важлива частина, 20% часу
4. **Рефакторинг** - ітеративний процес, 15% часу
5. **Архітектура** - базис для всього, 10% часу
