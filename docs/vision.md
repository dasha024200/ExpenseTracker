# ExpenseTracker - Vision Document

## Назва проєкту
ExpenseTracker - Система управління особистими витратами

## Опис проблеми
Більшість людей мають складність з відслідковуванням своїх денних витрат. Вони не знають, на що саме вони витрачають гроші, які категорії витрат найбільші, і як планувати свій бюджет на майбутнє. Існує потреба в простому та зручному інструменті для записування витрат з можливістю аналізу та категоризації.

## Цільові користувачі
1. **Приватні особи** - люди, які хочуть контролювати свої видатки
2. **Фінансово свідомі** - користувачі, які бажають планувати персональний бюджет
3. **Студенти** - люди з обмеженим бюджетом, які хочуть відслідковувати витрати

## Основні сценарії використання

### Сценарій 1: Додавання витрати
Користувач хоче записати витрату на обід. Він відкриває застосунок, вибирає опцію "Додати витрату", вводить суму (250 грн), категорію (Їжа), дату та описує витрату ("Обід в кафе"). Система зберігає витрату до памяті.

### Сценарій 2: Перегляд витрат за період
Користувач хоче побачити всі витрати за поточний місяць. Система виводить список усіх витрат за період, відсортованих за датою, з інформацією про категорію та суму.

### Сценарій 3: Отримання статистики по категоріям
Користувач хоче знати, скільки він витратив на кожну категорію. Система розраховує суму витрат за кожною категорією та виводить звіт.

## Нефункціональні вимоги

1. **Читабельність коду** - код повинен бути добре структурований, з понятливими назвами класів та методів, дотримуватися SOLID принципів
2. **Тестованість** - система повинна бути проектована так, щоб легко писати тести; мінімум 5 юніт-тестів на першій ітерації
3. **Відновлення після помилки** - застосунок повинен коректно обробляти помилки введення користувача та показувати зрозумілі повідомлення про помилки
4. **Простота розширення** - архітектура повинна дозволяти легко додавати нові функції без переписування існуючого коду (завдяки шаруватій архітектурі та інтерфейсам)

## Обмеження першої ітерації (Lab 34)

На цій ітерації ми не робимо:
- **Не реалізуємо persistence** - витрати зберігаються тільки в памяті сеансу, немає збереження у файл
- **Не реалізуємо фільтрацію та сортування** - це буде на Lab 35
- **Не реалізуємо веб-інтерфейс** - тільки консольний інтерфейс
- **Не реалізуємо бюджетні ліміти** - це буде розширенням на пізніших ітераціях
- **Не реалізуємо користувачів** - одиночний режим використання

## Архітектурна модель

```
Console UI -> Application Service -> Domain Model -> In-Memory Repository
```

Система розділена на чіткі шари:
- **Domain** - сутності (Expense, Category) з бізнес-правилами
- **Application** - сервіси для управління витратами
- **Infrastructure** - репозиторій (тимчасово в памяті)
- **Console** - користувацький інтерфейс
