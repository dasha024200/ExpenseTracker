## 2. Архітектурні рішення

### 2.1 Вибрана архітектура

**Шарова архітектура (Layered Architecture):**

```
Console Layer (UI)
     ↓
Application Layer (Business Logic)
     ↓
Domain Layer (Entities & Rules)
     ↓
Infrastructure Layer (Data Access)
```

### Причина вибору
- Чіткі межі відповідальності (SRP)
- Легко тестувати кожен шар окремо
- Можна замінювати реалізацію без змін інших шарів (OCP)
- Природна структура для SOLID применению

### 2.2 Патерни проектування

#### 1. **Repository Pattern**
- **Де:** Domain (інтерфейс) + Infrastructure (реалізація)
- **Чому:** Абстрагує джерело даних, дозволяє переключатися між JSON та БД
- **Вплив:** Значно спростив тестування та розширення

#### 2. **Service Layer Pattern**
- **Де:** Application層
- **Чому:** Централізує бізнес-логіку, відокремлює від UI та персистентності
- **Вплив:** Можна повторно використовувати сервіси в різних UI

#### 3. **Result Pattern**
- **Де:** Domain (Result<T> клас)
- **Чому:** Явна обробка помилок без винятків
- **Вплив:** Більш читабельний та контрольований код обробки помилок

#### 4. **Dependency Injection**
- **Де:** Console Program.cs
- **Чому:** Інверсія управління залежностями
- **Вплив:** Легко мокувати залежності у тестах

### 2.3 SOLID Principles

#### S - Single Responsibility
- `Expense` несе відповідальність тільки за сутність витрати
- `ExpenseService` - тільки за бізнес-логіку
- `FileSystemRepository` - тільки за файловий I/O
- ✅ **Результат:** Кожен клас має одну причину для змін

#### O - Open/Closed
- Інтерфейс `IExpenseRepository` дозволяє додавати нові типи персистентності
- Навички можна розширювати без змін сервісу
- ✅ **Результат:** Легко додавати нові можливості (SQL, MongoDB, Redis)

#### L - Liskov Substitution
- `InMemoryExpenseRepository` та `FileSystemExpenseRepository` комутабельні
- Сервіс працює однаково з обома реалізаціями
- ✅ **Результат:** Полімормфізм працює коректно

#### I - Interface Segregation
- `IExpenseRepository` містить тільки необхідні методи
- `IExpensePersistence` розділена для асинхронних операцій
- ✅ **Результат:** Клієнти не залежать від методів, які не використовують

#### D - Dependency Inversion
- Сервісми залежать від `IExpenseRepository`, не від конкретних класів
- Впорскування залежностей через конструктор
- ✅ **Результат:** Низька зв'язаність (loose coupling)

## 3. Технічні рішення

### 3.1 Вибір технологій

| Компонент | Технологія | Причина |
|-----------|-----------|---------|
| Мова | C# | Вимога курсу, добра типізація, SOLID підтримка |
| Framework | .NET 8 | Сучасна версія, кроссплатформне |
| Серіалізація | System.Text.Json | Вбудована, швидка, без залежностей |
| Тестування | xUnit | Сучасна альтернатива NUnit, добре інтегрується |
| Мокування | Moq | Популярна, добре задокументована |
| CI/CD | GitHub Actions | Вбудована в GitHub, free для public repos |
| Покриття | Coverlet | Cross-platform, хороша інтеграція з CI |

### 3.2 Обгрунтовані компроміси

#### 1. In-Memory категорії
- **Проблема:** Категорії можна розширювати, але вони не зберігаються
- **Причина:** Простота для курсового проекту, фокус на витратах
- **Альтернатива:** Категорії в JSON файлі (планується v1.1.0)
- **Вплив:** Категорії скидуються при перезапуску

#### 2. Синхронна JSON персистентність
- **Проблема:** Блокування потоку при завантаженні великих файлів
- **Причина:** Відсутність асинхронного I/O на Lab 34-35
- **Альтернатива:** async/await (планується v1.1.0)
- **Вплив:** На файлах > 10MB можлива затримка UI

#### 3. Console UI замість GUI
- **Проблема:** Текстовий інтерфейс менш привабливий
- **Причина:** Фокус на архітектурі, не на UI
- **Альтернатива:** WPF або Blazor (планується v1.1.0)
- **Вплив:** Менша привабливість для користувачів

#### 4. Result<T> замість Exception
- **Проблема:** Більше коду, менше привабливо для простих операцій
- **Причина:** Явна обробка помилок, функціональний стиль
- **Альтернатива:** Традиційні винятки
- **Вплив:** Більш контрольована обробка помилок

## 4. Рефакторинг та очищення коду

### 4.1 Code Smells, усунені на Lab 37

#### 1. Дублювання валідації
**Було:** Перевірка сум у декількох місцях
```csharp
// В Expense
if (amount <= 0) throw new Exception("...");

// В ExpenseService
if (amount <= 0) throw new Exception("...");

// В Console
if (amount <= 0) Console.WriteLine("...");
```

**Виправлено:** Централізована валідація
```csharp
// Тільки в конструкторі Expense
public Expense(int id, Category category, decimal amount, string description)
{
    if (amount <= 0)
        throw new ArgumentException("Сума повинна бути позитивною");
    // ...
}
```

#### 2. Магічні числа у меню
**Було:**
```csharp
while (true)
{
    int choice = int.Parse(Console.ReadLine());
    if (choice == 1) { ... }
    if (choice == 2) { ... }
    if (choice == 0) break;
}
```

**Виправлено:** Константи з enum
```csharp
public enum MenuOption
{
    Exit = 0,
    AddExpense = 1,
    ViewByMonth = 2,
    // ...
}
```

#### 3. Довгі методи
**Було:** `Program.Main()` з 200+ рядків
**Виправлено:** Розділено на відповідні класи:
- `Menu.cs` - меню опції
- `CommandHandler.cs` - обробка команд
- `UiHelper.cs` - форматування виводу

#### 4. Порожні блоки try-catch
**Було:**
```csharp
try 
{ 
    LoadData(); 
}
catch 
{ 
    // ignored
}
```

**Виправлено:** Явна обробка або логування
```csharp
try 
{ 
    LoadData(); 
}
catch (FileNotFoundException ex)
{
    Console.WriteLine($"Файл не знайдено: {ex.Message}");
}
```

#### 5. Невдалі імена змінних
**Було:** `e`, `d`, `temp`
**Виправлено:** `expense`, `description`, `categoryStatistics`
